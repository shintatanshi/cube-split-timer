Cube Split Timer F2L高速化まとめ
================================

作成目的
--------
AnalyzerのF2L解析を、既存手順DBベースの「FR / FL / BR / BL の最短順序比較」として高速化した内容をまとめる。


1. 体感速度まわり
-----------------
- 通常のF2L解析では、最初に「基本41ベース最短順序」だけを先に表示するようにした。
- 他の順序候補は「もっと見る」を押した時だけ追加計算するようにした。
- 解析ボタン自体は残し、押した結果としてDBベースF2Lだけが出る形に整理した。
- 未解決が残る場合だけ「未解決ペアを詳しく探索」で補助探索を走らせる形に分離した。


2. Workerまわり
---------------
- F2L解析をWeb Workerで実行するようにした。
- includeAllPlans を追加し、最短1件だけ計算する場合と24候補一覧を計算する場合を分離した。
- useLocalSearch を追加し、基本41DBのみ / 補助探索ありを分けた。


3. 探索量削減
-------------
- 24通りの順序配列を毎回作らず、F2L_ORDER_PERMUTATIONS として使い回すようにした。
- 最短候補探索では、すでに見つかった候補に勝てなさそうな順序を途中打ち切りするようにした。
- 候補ステップを全部配列に貯めてsortせず、最良候補だけ保持するようにした。


4. 手順DB・取り出し候補のキャッシュ
-----------------------------------
- Basic41手順をスロット別に事前展開してキャッシュするようにした。
- 手順文字列を毎回パースせず、moves / moveCount / score を保持するようにした。
- F2L取り出し候補も事前展開してキャッシュするようにした。
- getMoveDescriptor() もキャッシュ化した。


5. CubeState処理の軽量化
------------------------
- applyAlgorithm() を、1手ごとにCubeState全体cloneする方式から、最初に1回cloneして連続適用する方式に変更した。
- ピース検索を色配列の総当たりから piece.id 比較へ変更した。
- Cross完成判定を配列生成なしの直接判定に変更した。
- キャッシュキー用の状態署名から重いsortを削減した。


6. FastCubeState導入
--------------------
- F2L解析内部用に FastCubeState を追加した。
- 座標やステッカー面を Int8Array / Uint8Array で持つようにした。
- Basic41照合、F2Lペア完成判定、取り出し候補判定の一部をFastCubeStateベースに変更した。
- ステップ間ではFastCubeStateのまま渡し、必要な時だけCubeStateへ戻すようにした。
- F2L候補の中間状態をFastCubeStateのまま次ステップへ渡し、FastCubeState -> CubeState -> FastCubeState の往復を減らした。


7. UI整理
---------
- 最初は最短順序だけ表示するようにした。
- 他候補は「他の順序候補をもっと見る」で表示するようにした。
- 他候補計算中の表示を追加した。
- Cross変更や再解析時に、他候補の展開状態をリセットするようにした。
